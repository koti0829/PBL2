# Tuesday JSを使用したノベルゲーム共同開発ワークフロー

## 1. 使用ツール概要
* **ツール名:** Tuesday JS（日本語ビジュアル版）
* **作業URL:** https://kirilllive.github.io/tuesday-js/translate/ja_tuesday_visual.html
* **特徴:**
  * ブラウザ上で動作するインストール不要のノベルゲームエディタ。
  * フローチャート（ノード）を矢印で繋いでシナリオや分岐を作るノーコード仕様。
  * 制作データはすべて1つの「JSONファイル」としてエクスポート（保存）およびインポート（再開）する。
  * 画像や音声などの素材は、ローカルフォルダ内の相対パスで参照する。

## 2. 開発体制とアプローチ
* **開発人数:** 複数人（チーム開発）
* **管理ツール:** Git / GitHub（データのバックアップと共有のため）
* **採用する開発手法:** 「最初から完全にファイルを分ける方法」
  * **選定理由:** Tuesday JSは1つのJSONファイルに全データが書き出されるため、全員で同じファイルを同時に編集すると、Git上で「衝突（コンフリクト）」が起きた際にJSONコードが壊れるリスクがある。そのため、最初から担当ごとに「別々のJSONファイル」として作成し、最後に手動で1つに合体させる安全な手法を採用する。

## 3. 事前準備と最重要ルール

### ルール①：担当ファイル名と「シーン名」の命名規則
各自がTuesday JSで新規作成するファイル名と、ツール内で作成する「シーン名（ブロック名）」の頭に、必ず担当の識別子をつける（名前の重複によるデータ破損を防ぐため）。

**確定した担当表:**

| 担当 | 学年 | ファイル名 | ブロック名の接頭辞 |
|---|---|---|---|
| 住川 | 1年生編 | `ch1.json` | `ch1_scene*` |
| 戸井 | 2年生編 | `ch2.json` | `ch2_scene*` |
| 臼井 | 3年生編 | `ch3.json` | `ch3_scene*`（岡/荒井/五島ルートで`ch3_oka_*`/`ch3_arai_*`/`ch3_goto_*`） |
| 萩原 | 4年生編 | `ch4.json` | `ch4_oka_*`/`ch4_arai_*`/`ch4_goto_*` |

学年をまたぐ遷移（1年→2年、2年→3年、3年→4年）はあらかじめ`go_to`で次章の最初のブロック名（`ch2_scene1`など）に配線済みなので、合体後に手動でリンクを繋ぎ直す必要はない。

### ルール②：素材フォルダ構造の一致
各自のPC内のフォルダ構成、およびファイル名を完全に一致させ、Gitで共有する。

```text
professor-galgame/          (ゲームのルートフォルダ)
├── ch1.json                 (Aさんのファイル)
├── ch2.json                 (Bさんのファイル)
└── data/                    (★共通の素材フォルダ)
    ├── bg/                  (背景画像を入れる)
    └── chara/               (立ち絵を入れる)
```

新しい画像を追加する場合は、全員が同じフォルダ（例: `data/chara/hero.png`）に同じ名前で配置してGit経由で同期する。

## 4. 日々の作業フロー（各自のルーティン）
ファイル名が異なるため、誰がいつ作業してGitHubにPushしても、他の人のデータを上書きするリスクはない。
1. **作業開始時:** Gitツール（GitHub Desktop等を推奨）で最新状態を取得する（Pull）。
2. **インポート:** ブラウザでTuesday JSを開き、自分の担当ファイル（例: `ch1.json`）をインポートする。
3. **制作・保存:** シナリオや演出を制作したら、エディタからプロジェクトをエクスポートし、自分の担当ファイルに上書き保存する。
4. **作業終了時:** 変更した自分のJSONファイルと、新しく追加した画像などの素材をGitでGitHubに送る（Push）。

## 5. 最後の統合（ファイルを1つに合体させる手順）
ゲームの全章が完成したら、テキストエディタ（VS Codeなど）を使用してバラバラのファイルを1つにドッキングする。

**注意:** Tuesday JSの実際のエクスポート形式は`"scenes": [...]`という配列ではない。実際は次のような構造になっている。

```json
{
  "parameters": { "characters": {...}, "variables": {...}, "launch_story": "...", ... },
  "blocks": { "ブロック名": ["x座標", "y座標", "block", false], ... },
  "ch1_scene1": [ { "background_image": {...}, "dialogs": [ ...ノード配列... ] } ],
  "ch1_scene2": [ ... ]
}
```

つまり各シーン（ブロック）は`"scenes"`配列の中身ではなく、JSONの**トップレベルにある動的なキー**として1つずつ格納されている。`blocks`はエディタ上の位置情報レジストリ。

【合体の具体的な手順】
1. 4人分の`ch1.json`〜`ch4.json`をテキストエディタで開く。
2. ベースとなるファイル（例: `ch1.json`）に、他の3ファイルの**`parameters`と`blocks`以外のトップレベルキー**（`ch2_scene1`, `ch3_oka_scene2`, `ch4_arai_scene1`など、全ブロック）をすべてコピーして追加する。
3. `blocks`オブジェクトも同様に、他3ファイルの`blocks`の中身（位置情報）をベースファイルの`blocks`にマージする。
4. `parameters`は1つだけ残せばよい（全員が同じ共有テンプレートを使っているため、キャラクター定義・変数定義は完全に一致しているはず）。
5. 保存して、合体させたファイルをTuesday JSにインポートし、崩れずに読み込めるか確認する。
6. 最後にエディタから `game_complete.json` などの名前でエクスポートすれば、1つの大きなゲームファイルが完成する。

**重要:** `parameters.characters`と`parameters.variables`は4人全員のファイルで完全に一致させること。1つでも異なるファイルがあると、そのファイルの`parameters`を採用したときに他の章のキャラクター/変数が欠落する。

## 6. 分岐ロジック（ルート決定・エンディング分岐）の実装方法
3年生編の「好感度が最も高い教授のルートに分岐」、4年生編の「エンディング分岐（告白エンド/卒業エンド）」は、Tuesday JSの`js`ノードと`legacy_choice`ノードを組み合わせて実装している。ここは通常のノーコード操作では組めないため、変更する場合は以下の実際の仕様に注意すること（Tuesday JS本体のソースコードで確認済み）。

* **`legacy_choice`ノード:** `[[変数名, 演算子, 比較値, 遷移先ブロック名], ...]`という4要素配列のリストのみが有効。演算子は `">"` / `"<"` / `"="` の3つだけ（`">="`のような複合演算子は無い）。配列を先頭から評価し、条件が一致した最初のエントリの`go_to`に遷移する。**「以外は全部ここへ」という汎用の`else`は無いので**、必ず最後に確実に真になる条件（例: `[["variable", ">", -1000, "fallback"]]`）を入れて全パターンを網羅すること。
* **`js`ノード:** `{"js": "任意のJSコード文字列"}`。`story_json.parameters.variables.変数名`で変数の読み書きができる。クリック不要で自動的に次のノードへ進む。
* **選択肢の`variables`（好感度加算など）:** 演算子は`"add"`と`"set"`のみ有効（`"remove"`は存在しない）。減算したい場合は`["変数名", "add", -5]`のように負の数を渡す。
